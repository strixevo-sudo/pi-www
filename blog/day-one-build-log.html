Day One - The Build Log

23 Mar 20268 min readby STRIXEVO

Every project has a day one. The day where nothing exists yet and by the time you go to sleep something does. This was mine. Here's everything that happened - from blank files to a live, hardened, self-hosted website - in a single sitting.

// What this is A full dev log of day one building str1x3vo.xyz - every feature added, every problem hit, every command run. Raw and unfiltered.

The starting point

The site already existed in a basic form - HTML files sitting in /var/www/html on a Kali Linux VM, served by Nginx, tunnelled to the internet via Cloudflare because my ISP uses CGNAT and port forwarding isn't an option. The bones were there. Everything else needed building.

The first job was getting the files actually looking like a proper website. That meant wiring up a shared style.css so every page pulls from one stylesheet instead of having hundreds of lines of inline CSS copy-pasted everywhere. Cleaner, faster, easier to maintain. The kind of thing you don't bother doing when you're hacking something together at midnight but absolutely have to sort before it becomes a problem.

JavaScript - bringing it to life

The CSS already had scroll reveal classes defined - .reveal, .reveal-left, .reveal-right - but nothing was triggering them. No JavaScript meant no animations. Sections just sat there, static, like a poster instead of a website.

Built out main.js from scratch. Scroll reveal using IntersectionObserver - sections and cards fade and slide in as you scroll down, grid items staggering in one by one. Nav scroll effect that darkens the background once you're past the hero. Active link highlighting as you move between sections. And the one that took the most tuning - a typing effect on the hero title that cycles through different lines, deletes them character by character, and loops.

Clean URLs

Nobody wants to see .html in a URL. It looks unfinished. One Nginx config change fixes it - try_files $uri $uri.html $uri/ =404 in the location block. Now /how-i-set-up-my-website serves the right file without the extension showing. Every blog post got renamed and re-linked to match.

The socials and downloads

Wired up the social cards properly - YouTube, Twitch and Discord all pointing to the right places. Took longer than it should have thanks to some creative sed command disasters that kept swapping the links around. Eventually brute forced it with Python string replacement and it's been fine ever since.

Downloads section pointing to GitHub in the meantime. Repos coming once there's actually something worth releasing.

PC specs - now with icons

Added component icons to the spec table - CPU, GPU, motherboard, RAM, SSD all have matching icons sitting next to the spec details. Small thing, makes a big difference. The GPU icon needed upsizing because the image had more padding around it than the others - minor but the kind of detail that matters when you're looking at a table of hardware.

Happy birthday Sketchy Koko

Took a detour to build a birthday banner for a mate - Sketchy Koko. Gold banner pinned to the bottom of the screen, confetti raining down for 12 seconds, a direct link to his Twitch, and a close button. The whole thing checks today's date on load and only shows on the 23rd of March. Set it to fire every year so it comes back automatically.

Cheers for the Pi, by the way.

Security - locking it down

A self-hosted site sitting on a residential connection needs hardening. Spent the last chunk of the session on this.

Fail2ban first - installed, configured, three jails running. SSH gets a 24 hour ban after 3 failed login attempts. Nginx bot scanning and auth brute forcing both covered. Anyone trying to hammer the server gets cut off fast.

sudo fail2ban-client status
|- Number of jail: 3
`- Jail list: nginx-botsearch, nginx-http-auth, sshd

Nginx hardening next - version number hidden, request size capped, timeout protection, buffer overflow prevention, hidden files blocked, common exploit scanner patterns returning 444, sensitive file extensions denied. The config tested clean.

SSH key authentication - generated an ed25519 key pair on Windows, public key dropped into ~/.ssh/authorized_keys on Kali. Password login getting disabled once the key auth is confirmed working. After that, no password means nothing to brute force.

What's next

Finish the SSH key setup and disable password login. Get the Raspberry Pi configured and migrate hosting off the gaming PC. Add more blog posts. Build out the downloads section with actual releases. Keep going.

Not bad for a day's work.
