What a Real Design Review Changed on My Website

25 APR 2026
6 MIN READ
BY STR1X3VO

I wanted a real opinion on the site, not just me staring at it until everything looked normal because I built it. So I had an outside reviewer go through str1x3vo.xyz, write up the problems, then post the whole thing publicly as a GitHub issue. That turned out to be exactly what the site needed.

// What this covers An external design audit, the fixes that actually mattered, the infrastructure problem hiding underneath the design notes, and why this site is better because someone else picked holes in it.

Why I wanted outside eyes on it

When you build your own website, you stop seeing the obvious stuff. You know where every section is, you know what every animation is supposed to do, and you stop noticing the parts that would feel awkward to someone landing on it for the first time.

That is exactly where an external review helps. Not because every comment is automatically right, but because someone coming in cold spots the things you have become blind to.

In my case that review lived in GitHub issue #1. It started as a baseline audit, then got re-run after the first modernization pass, then got checked again after the follow-up fixes. Proper paper trail. Nothing vague. Everything tied to actual files and actual live behavior.

What got called out

The first pass was useful because it split the feedback into real problems, smaller polish issues, and style calls. The biggest technical ones were the kind of things that matter more than they look on paper:

Content hidden without JavaScript. The reveal animations looked fine when everything loaded normally, but if JavaScript failed then large sections of the page were effectively invisible. That is not a design problem anymore. That is just broken.

Touch targets and readability. The mobile nav already got fixed in the main redesign, but the remaining audit still pushed on touch target size, post heading hierarchy, and long-form reading rhythm. Those are the details that separate something that looks nice from something that actually feels finished.

Font loading and first paint. The homepage was loading IBM Plex Serif even though it did not use it, fonts were coming in through CSS @import, and the hero title briefly rendered as STR1 before the rest of the name typed in. Small stuff individually. Collectively, they make the site feel less sharp than it should.

The fixes that actually mattered

The follow-up pass was not a second redesign. It was a cleanup pass. The goal was to keep the better modernized look, then remove the remaining weak points without flattening the site into something generic.

document.documentElement.classList.add('js')
.js .reveal { opacity: 0; }
--font-display: 'Orbitron', 'Arial Black', sans-serif;
style.css?v=7
main.js?v=6

That one group of changes covered a lot of the audit in practice. The page now defaults visible unless JavaScript explicitly enables the reveal behavior. Post headings were given real weight with a top rule and larger type. Nav links were bumped to a proper touch size. The hero paints X3VO immediately instead of waiting for the typing effect to catch up.

I also moved Google Fonts out of CSS and into the HTML heads where they belong. Homepage only loads the faces it actually uses. Article pages still get IBM Plex Serif because that is where it earns its keep.

The hidden problem was not design

The most important fix from that whole review was not visual at all. The audit spotted that GitHub had the newer site, while the live Pi was still serving old files. That looked like a deployment mistake on the surface. Underneath, it turned into a repo and auth problem.

The Pi checkout needed repairing, and the Git remote needed to stop depending on a broken HTTPS flow that would have kept nagging for credentials. So the site did not just get prettier. The deployment path got cleaned up too, with the Pi repo repaired and SSH-based GitHub access restored properly.

// The engineering bit Fixing one ugly button is fine. Fixing the reason your server falls out of sync with GitHub is better. That is the kind of boring fix that saves you time every single time you ship after it.

What I did not change

This part matters as much as the fixes. I did not blindly accept every suggestion. The stacked STR1 / X3VO hero stayed once the first-paint problem was fixed, because after that it was clearly a style choice, not a rendering artifact. The featured post treatment on the blog grid stayed too, because it is one of the strongest bits of hierarchy on the homepage.

That is how review should work. You take the useful pressure. You fix what is genuinely weak. You keep the parts that still serve the identity of the site.

The result

The re-check came back clean. No remaining defects from that audit pass. The reviewer signed off on the live site, not just the repo. The scores moved up again, but more importantly the site now feels tighter in the places people hit first: first paint, mobile interaction, long-form reading, and deployment reliability.

I also versioned the whole thing properly. There is now a clean chain of tags for the site state before the rework, after the modernization pass, and after the audit-fix pass. That makes rollback and comparison dead simple instead of guesswork.

So yeah, this post is basically a thank you note to external criticism. It is very easy to say you want feedback. It is harder to put it in public, wire it to the real code, and work through it item by item. But that is exactly what made the site better.

// Final status External review. Two fix passes. Live Pi back in sync. Version tags in place. Site went from "good enough" to something a lot more deliberate.
